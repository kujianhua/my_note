# 规范化

利用githooks，在开发同学提交代码的时，统一代码风格，做代码校验，杜绝不满足规则的代码提交上云；

# git hooks

git的钩子函数，通常用来做：

  • 提交前做代码检测
  • 提交时格式化代码，做代码美化兜底
  • 提交后做自动化部署
  • ...

常用githook：

  • pre-commit
  • commit-msg

# 代码质量检查

  • jslint
  • jshint
  • eslint
  • prettier
  • lint-staged [一个在git暂存区上运行linters的工具。]
  • husky [一个让配置 git 钩子变得更简单的工具。]

# husky

Git hooks made easy（让git hooks变得简单）

新版本，已不要再操作.git/hooks文件夹里的文件

tyarn add husky -D      // 安装
tyarn husky install     // 项目根目录创建.husky文件夹，并指定为githooks目录,可提交到远程仓库
tyarn add lint-staged -D
tyarn add eslint prettier -D
tyarn husky add .husky/pre-commit 'yarn lint-staged --allow-empty "$1"'  // 创建钩子
tyarn husky add .husky/commit-msg 'yarn commitlint --edit "$1"'  // 创建钩子
tyarn add @commitlint/cli @commitlint/config-conventional -D
echo "module.exports = {extends: ['@commitlint/config-conventional']}" > commitlint.config.js

npx husky-init && yarn // 一键安装与配置

package.json

{
  "scripts": {
    "prepare": "husky install"
  },
  "lint-staged": {
    "src/**/*.{js,jsx,ts,tsx,json}": [
      "prettier --write",
      "eslint",
      "git add"
    ]
}
}
// prepare脚本会在npm install（不带参数）之后自动执行。也就是说当我们执行npm install安装完项目依赖后会执行 husky install命令

commitlint.config.js

 module.exports = {
  extends: ['@commitlint/config-conventional'],
  // 以下时我们自定义的规则
  rules: {
    'type-enum': [
      2,
      'always',
      [
        'bug', // 此项特别针对bug号，用于向测试反馈bug列表的bug修改情况
        'feat', // 新功能（feature）
        'fix', // 修补bug
        'docs', // 文档（documentation）
        'style', // 格式（不影响代码运行的变动）
        'refactor', // 重构（即不是新增功能，也不是修改bug的代码变动）
        'test', // 增加测试
        'chore', // 构建过程或辅助工具的变动
        'revert', // feat(pencil): add ‘graphiteWidth’ option (撤销之前的commit)
        'merge', // 合并分支， 例如： merge（前端页面）： feature-xxxx修改线程地址
      ],
    ],
  },
};

Antd Pro 自动生成api 依赖openAPI插件（自带）

然后再 config/config.ts 中配置 openAPI 的相关配置。

openAPI: [
  {
   // 可引入定制化request
   requestLibPath: "import { request } from 'umi'",
   // 后端同学的接口文档地址（swagger）
   schemaPath: "http://192.168.30.116:9500/v2/api-docs",
   // 自动生成mock数据 
   mock: false,
   // 文件夹名称
  //  projectName: 'branch'
 },
]

执行命令

npm run openapi

API接口用法说明

  • requestTable 用于table类接口调用，已封装通用入参、出参，定制化入参需特殊处理，出参为前后端约定固定数据结构，如有改动及时沟通；

  • request 用于基础接口调用

  • 所有接口可直接从openAPI自动生成的API复制过来使用，无需改动，以及对应的type数据结构

  • isSuccess 业务代码内判断接口返回是否成功

    const res:ApiResponseXxx = await xxxXxx(params);
      if(isSuccess(res)){
      // do something 接口返回成功
      }
      // 接口返回失败通常会统一拦截处理，有特殊要求可在此操作

  • 后端返回的对象({})、数组([]),都需要在取数据的时候做为空处理,如下：

    const a = res.data || {};
const {a = '' ,b = []} = res.data;
const a = res.data || [];
// 这种写法在变量声明是依赖类型推导，

建议在声明时，明确数据类型 openAPI会有生成；
const res: PageResultAppDTO_ = await xxXxxx(params);
const { b: PageResultAppDTO_ = {} } = res.data;
const c: PageResultAppDTO_ = res.data || {};

# 命令

  • vim 打开文件
    1. i 进行插入操作
    2. control + c 结束插入
    3. :qa 保存并且退出
    4. :wq 强行保存退出
  • ll -l@ 查看文件列表
    1. -l 查看详细附加内容
    2. -l@ 查看@的内容
  • xattr -c fileName 去掉文件权限后面的@，文件被非默认应用打开修改后，会出现此问题
  • mv 移动文件
    • mv file dir 文件移动到文件夹里
    • mv dir dir2 文件夹移动到文件夹，dir2不存在则为改名操作
    • mv file file2 file3 dir 多个文件移动到文件夹，文件夹不存在则报错
  • rm 删除文件/目录
  • mkdir 创建文件夹
  • pwd 查看当前目录路径
  • cp 拷贝
