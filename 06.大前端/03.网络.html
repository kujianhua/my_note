# 网络错误

  • 3xx （重定向）
  • 400 （错误请求） 服务器不理解请求的语法。（解决办法传参格式不正确）

# 解决思路

  • 网络请求是否准确
    • host（http/https、地址、端口号）
    • API （拼写、大小写、接口更新未及时同步、自动拼接时多余的‘/’）
    • method （POST/GET）
    • 参数 （参数类型 json/form-data、参数数据结构、必填参数缺失、参数对应不上/拼写错误）
    • header （参数不支持中文，请求会直接报错，network 都看不到请求信息，可通过编码解决）
  • 请求是否成功发送
    • 网页发送状态
    • node 服务接收网页请求
    • node 服务处理请求
    • node 服务发送请求到后端服务
  • 请求是否成功返回
    • node 服务接收后端服务返回
    • node 服务处理返回数据
    • node 服务返回数据到网页
  • 返回数据结构是否准确

# 同源&同站

源（origin）= 协议（scheme）+ 主机名（hostname）+ 端口号（port）

站（site）= eTLD+1 (不看协议、端口、子域名)
  • TLD (表示顶级域名，例如 .com、.org、.cn 等等)
  • TLD+1 (表示顶级域名和它前面二级域名的组合， 例如: https://www.example.com )
  • eTLD（有效顶级域名，由 Mozilla 维护在公共后缀列表（Public Suffix List）中，例如:.io、.com.cn）
  • eTLD+1 (有效顶级域名和它前面二级域名的组合， 例如: https://www.example.com.cn )

# Referer

请求头包含了当前请求页面的来源页面的地址，即表示当前页面是通过此来源页面里的链接进入的。服务端一般使用 Referer 请求头识别访问来源，可能会以此进行统计分析、日志记录以及缓存优化等。

在以下两种情况下，Referer 不会被发送：

  • 来源页面采用的协议为表示本地文件的 "file" 或者 "data" URI；
  • 当前请求页面采用的是非安全协议，而来源页面采用的是安全协议（HTTPS）。

# Origin

请求首部字段 Origin 指示了请求来自于哪个站点。该字段仅指示服务器名称，并不包含任何路径信息。该首部用于 CORS 请求或者 POST 请求。除了不包含路径信息，该字段与 Referer 首部字段相似。

同源请求：POST、OPTIONS、PUT、PATCH 和 DELETE 请求都会添加 Origin 请求头，GET 或 HEAD 请求不会添加 Origin 请求头。 跨域请求：所有跨域请求(CORS)都会添加 Origin 请求头。 来源页面采用的协议为表示本地文件的 "file" 或者 "data" URI, origin:null

# UDP

  • 支持一对一，一对多，多对一和多对多的交互通信
  • 无连接
  • 可靠性存在缺陷
  • 一个包一个包的发送，是有边界的，但可能会丢包和乱序。
  • 低数据量传输速度较 TCP 快 14%，超过 1M 数据快不了多少
  • 应用：包总量较少的通信，如 DNS 、SNMP 等；视频、音频等多媒体通信；广播通信；
